Section 01

전체 아키텍처 개요

이 서버는 Windows IOCP(I/O Completion Port) 기반의 비동기 TCP 게임 서버입니다. 클라이언트로부터 게임 데이터 패킷을 수신하여 Redis에 캐싱하고, 주기적으로 MySQL에 영속화합니다. 별도의 Node.js 관리 서버가 동일한 Redis/MySQL에 접근하여 HTTP REST API를 제공합니다.

Section 02

핵심 컴포넌트 상세 설명

NETWORK
IocpCore
IocpCore.h / IocpCore.cpp

서버의 비동기 I/O 엔진. IOCP 핸들을 생성하고, hardware_concurrency × 2개의 워커 스레드를 운용한다.

워커 스레드 루프

while (m_isRunning) {
    GetQueuedCompletionStatus(m_hIocp, &bytes, &key, &overlapped, INFINITE);

    ExOverlapped* exOver = reinterpret_cast<ExOverlapped*>(overlapped);
    switch (exOver->io_type) {
    case IO_TYPE::ACCEPT:  → Acceptor::OnAcceptCompleted()
    case IO_TYPE::RECV:    → Session::OnRecvCompleted() → PacketHandler
    case IO_TYPE::SEND:    → Session::SendCompleted()
    }
}
  • INFINITE 대기 — CPU 점유 없이 완료 이벤트만 처리
  • 종료 시 PostQueuedCompletionStatus(0)으로 각 스레드 깨움

Acceptor
Acceptor.h / Acceptor.cpp

AcceptEx 기반 비동기 Accept. 시작 시 256개의 AcceptEx를 사전 예약하여 동시 연결 급증에 대응한다.

// 초기화 시
for (int i = 0; i < PENDING_ACCEPTS; ++i)   // 256개
    RegisterAccept();

// 완료 콜백
void OnAcceptCompleted(AcceptOverlapped* over) {
    setsockopt(SO_UPDATE_ACCEPT_CONTEXT);    // 필수
    auto session = make_shared<Session>(over->clientSock);
    CreateIoCompletionPort(clientSock, iocp, session.get(), 0);
    SessionManager::AddSession(session);
    session->PostRecv();
    RegisterAccept();                        // 즉시 1개 보충
}

NETWORK
Session
Session.h / Session.cpp

클라이언트 연결 1개를 표현하는 클래스. enable_shared_from_this로 IOCP Overlapped 내에서 수명을 관리한다.

수신 파이프라인

// WSARecv 완료 시
void OnRecvCompleted(int bytes) {
    m_ringBuffer.Write(m_recvBuffer.data(), bytes);   // 원형 버퍼에 적재
    while (auto pkt = m_ringBuffer.TryAssemblePacket())
        PacketHandler::Handle(shared_from_this(), *pkt); // 완성된 패킷 처리
}

송신 파이프라인

void PostSend(char* data, size_t size) {
    m_sendQueue.push(vector<char>(data, data+size));
    if (m_isSending.exchange(true) == false)
        RegisterSend();   // 최초 한 번만 WSASend 시작
}
void SendCompleted() {
    m_sendQueue.pop();
    RegisterSend();       // 큐가 빌 때까지 연속 전송
}
  • m_recvBuffer : 8KB 직접 수신 버퍼
  • m_ringBuffer : 16KB 원형 버퍼 (패킷 경계 조립)
  • m_sendQueue + m_isSending : 전송 직렬화 (mutex + atomic)
  • keepAlive : OVERLAPPED가 완료될 때까지 Session 수명 보장
BUFFER
RingBuffer
RingBuffer.h / RingBuffer.cpp

TCP는 스트림이므로 여러 recv 결과를 합쳐야 완전한 패킷이 된다. RingBuffer는 wrap-around 원형 버퍼로 이를 처리한다.

// 패킷 조립 로직
optional<vector<char>> TryAssemblePacket() {
    if (dataSize < sizeof(PacketHeader)) return nullopt; // 헤더 미도착
    PacketHeader header;
    Peek(&header, sizeof(PacketHeader));                 // 헤더만 엿보기
    if (dataSize < header.size) return nullopt;          // 바디 미도착
    vector<char> pkt(header.size);
    Read(pkt.data(), header.size);                       // 완성 패킷 추출
    return pkt;
}
wrap-around 처리
버퍼 끝에 걸쳐있는 데이터는 memcpy를 두 번으로 나눠 처리 (끝 + 처음). 읽기/쓰기 포인터는 % BUFFER_SIZE로 순환.

// 커넥션 대기 — 5초 타임아웃
MYSQL* GetConnection() {
    unique_lock lock(m_poolLock);
    if (!m_poolCondition.wait_for(lock, 5s,
            [this]{ return !m_connectionPool.empty(); })) {
        LogError("커넥션 획득 타임아웃");
        return nullptr;
    }
    return m_connectionPool.back(); // pop
}
void ReturnConnection(MYSQL* conn) {
    { lock_guard lock(m_poolLock); m_connectionPool.push_back(conn); }
    m_poolCondition.notify_one();   // 대기 스레드 깨움
}

vector<uint64_t> GetAllCachedCharacterIds() {
    long long cursor = 0;

측정 환경 주의사항
로컬루프백(127.0.0.1) 기준. send()는 커널 송신 버퍼 기록 시점 기준이며, 서버의 실제 패킷 처리 완료 시점이 아님. WAN 환경에서는 레이턴시 및 TCP 재전송으로 수치가 낮아짐.

발견 및 수정한 버그 — 디버깅 과정

1
listen backlog 부족 — 접속 성공률 40% listen(sock, 10)으로 TCP 대기열이 10개 한계. 1000 SYN 동시 도착 시 90%가 OS 레벨에서 드롭됨. → SOMAXCONN으로 변경, AcceptEx 사전 예약 64 → 256개
2
IOCP 워커 스레드 전체 블로킹 — 접속 성공률 61% OnCharacterStat에서 DBManager::UpdateCharacterStat()를 IOCP 워커 스레드에서 동기 호출. DB 풀 10개가 모두 점유되면 나머지 워커가 condition_variable에서 최대 5초 대기, 그 사이 AcceptEx 완료 이벤트 처리 불가 → listen backlog 재포화. → RedisManager::SetCharacterStat()으로 교체. MySQL 반영은 SyncWorker에 위임.
3
RECV / AcceptEx 이벤트 경합 — 접속 성공률 61% 지속 연결 직후 패킷 전송 시작으로 RECV 완료 이벤트가 AcceptEx 완료 이벤트보다 압도적으로 많아짐. 모든 워커가 RECV 처리에 집중, AcceptEx 완료 처리 지연 → SYN 드롭 재발. → std::latch로 연결 페이즈(AcceptEx만 존재)와 전송 페이즈를 분리.
4
1스레드=1소켓 구조 한계 — 1K → 10K 확장 불가 10,000 스레드 = 10GB 가상 스택 메모리 + OS 스케줄러 과부하. → 스레드 풀 32개로 전환. 각 스레드가 ~313 소켓을 순차 관리.
5
sleep 누적 버그 — 예상 300ms → 실제 95초 순차 루프 내에서 sleep_for(batchIndex × 3ms)를 클라이언트마다 호출. worker 31 기준: 100개 × 291ms = 29초씩 4배치 = ~92초 낭비. → sleep_until(testStart + batchIndex × 3ms)로 절대시각 기준 변경. 300ms 내 완료.

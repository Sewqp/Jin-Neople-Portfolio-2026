Section 01

전체 아키텍처 개요

Windows IOCP 기반 비동기 TCP 게임 서버. 클라이언트로부터 게임 데이터 패킷을 수신하여 Redis에 캐싱하고, 주기적으로 MySQL에 영속화한다. 별도의 Node.js 관리 서버가 HTTP REST API를 제공한다.

Section 02

핵심 컴포넌트 상세 설명

NETWORK
IocpCore
IocpCore.h / IocpCore.cpp

서버의 비동기 I/O 엔진. IOCP 핸들을 생성하고 hardware_concurrency × 2개의 워커 스레드를 운용한다.

while (m_isRunning) {
    GetQueuedCompletionStatus(m_hIocp, &bytes, &key, &overlapped, INFINITE);

    ExOverlapped* exOver = reinterpret_cast<ExOverlapped*>(overlapped);
    switch (exOver->io_type) {
    case IO_TYPE::ACCEPT:  Acceptor::OnAcceptCompleted();  break;
    case IO_TYPE::RECV:    session->OnRecvCompleted();     break;
    case IO_TYPE::SEND:    session->SendCompleted();       break;
    }
}
  • INFINITE 대기 — 완료 이벤트 없을 때 CPU 점유 없음
  • 종료 시 PostQueuedCompletionStatus(0) × N으로 각 워커 스레드 깨움
NETWORK
Acceptor
Acceptor.h / Acceptor.cpp

AcceptEx 기반 비동기 Accept. 시작 시 256개의 AcceptEx를 사전 예약하여 동시 연결 급증에 대응한다. 완료 후 즉시 1개를 보충하여 풀 유지.

// 완료 콜백
void OnAcceptCompleted(AcceptOverlapped* over) {
    setsockopt(SO_UPDATE_ACCEPT_CONTEXT);       // 필수
    auto session = make_shared<Session>(sock);
    CreateIoCompletionPort(sock, iocp, session.get(), 0);
    SessionManager::AddSession(session);
    session->PostRecv();
    RegisterAccept();                           // 슬롯 즉시 보충
}

Session
Session.h / Session.cpp

클라이언트 연결 1개를 표현. enable_shared_from_this로 IOCP OVERLAPPED 내에서 수명을 관리한다.

수신 파이프라인

void OnRecvCompleted(int bytes) {
    m_ringBuffer.Write(m_recvBuffer.data(), bytes);
    while (auto pkt = m_ringBuffer.TryAssemblePacket())
        PacketHandler::Handle(shared_from_this(), *pkt);
    PostRecv(); // 다음 WSARecv 즉시 재등록
}

송신 직렬화 (mutex + atomic)

void PostSend(char* data, size_t size) {
    m_sendQueue.push(vector<char>(data, data+size));
    if (m_isSending.exchange(true) == false)
        RegisterSend();   // 최초 한 번만 WSASend 시작
}
void SendCompleted() {
    m_sendQueue.pop();
    RegisterSend();       // 큐가 빌 때까지 연속 전송
}
BUFFER
RingBuffer
RingBuffer.h / RingBuffer.cpp

TCP 스트림에서 완전한 패킷을 조립하기 위한 16KB wrap-around 원형 버퍼. 패킷 경계가 버퍼 끝을 넘어도 memcpy를 두 번 나눠 처리한다.

optional<vector<char>> TryAssemblePacket() {
    if (dataSize < sizeof(PacketHeader)) return nullopt; // 헤더 미도착
    PacketHeader hdr;
    Peek(&hdr, sizeof(PacketHeader));                    // 읽기 포인터 유지
    if (dataSize < hdr.size)         return nullopt;    // 바디 미도착
    vector<char> pkt(hdr.size);
    Read(pkt.data(), hdr.size);                          // 완성 패킷 추출
    return pkt;
}

MYSQL* GetConnection() {
    unique_lock lock(m_poolLock);
    if (!m_poolCondition.wait_for(lock, 5s,
            [this]{ return !m_connectionPool.empty(); })) {
        LogError("커넥션 획득 타임아웃");
        return nullptr;

    return m_connectionPool.back();
}

void FlushAll() {

Section 03

패킷 데이터 흐름

수신 파이프라인

1
클라이언트 send() OS TCP 스택이 데이터를 수신 버퍼에 적재. IOCP가 WSARecv 완료 이벤트를 큐에 발행.
2
IocpCore 워커 스레드 — RECV 완료 처리 Session::OnRecvCompleted(bytes) 호출. 수신된 raw 바이트를 RingBuffer에 Write.
3
RingBuffer::TryAssemblePacket() 헤더의 size 필드를 확인하여 완전한 패킷이 모였는지 검사. 미완성이면 nullopt 반환 후 다음 recv 대기.
4
PacketHandler::Handle() PacketID 기준 switch-case 분기.
 · CHARACTER_STAT_INFO(1003) → RedisManager::SetCharacterStat() (빠름, <0.1ms)
 · CHARACTER_INFO(1002) / INVENTORY(1006) / AUCTION(1007) → DBManager CRUD
5
Session::PostRecv() 재등록 다음 데이터를 받기 위해 즉시 WSARecv 재등록.

Redis → MySQL 동기화 (SyncWorker, 30초 주기)

1
SCAN cursor=0, MATCH character:stat:*, COUNT 100 KEYS 명령 사용 금지 (Redis 블로킹). SCAN으로 cursor=0이 될 때까지 반복 열거.
2
RedisManager::GetCharacterStat(id) HGETALL character:stat:{id} 로 각 캐릭터 스탯 조회.
3
DBManager::UpdateCharacterStat(stat) Prepared Statement로 character_stat 테이블 UPDATE. 성공/실패 카운트 집계 후 로그 출력.

측정 환경
로컬루프백(127.0.0.1) 기준. send()는 커널 송신 버퍼 기록 시점 기준 — 서버의 실제 패킷 처리 완료 시점이 아님. WAN 환경에서는 레이턴시 및 TCP 재전송으로 수치가 낮아짐.

발견 및 수정한 버그

1
listen backlog 부족 listen(sock, 10) — 1,000 SYN 동시 도착 시 OS가 TCP 레벨에서 90% 드롭. 3초 후 SYN 재전송 포기로 접속 실패.
→ SOMAXCONN으로 변경
2
IOCP 워커 스레드 전체 블로킹 OnCharacterStat에서 MySQL 동기 호출 → DB 풀 10개 소진 → 나머지 워커 최대 5초 대기 → AcceptEx 처리 중단 → listen backlog 재포화.
→ Redis Write-Through로 교체. MySQL 반영은 SyncWorker에 위임.
3
RECV / AcceptEx 이벤트 경합 연결 직후 패킷 전송 → 수만 건의 RECV 완료 이벤트가 AcceptEx 완료 이벤트를 IOCP 큐에서 밀어냄 → 잔여 연결 처리 지연.
→ std::latch로 연결 페이즈(AcceptEx만 존재)와 전송 페이즈 분리.
4
1스레드=1소켓 구조 한계 10,000 스레드 = 10GB 가상 스택 메모리 + 스케줄러 과부하.
→ 스레드 풀 32개 전환. 각 스레드가 ~313개 소켓을 순차 관리.
5
sleep 누적 버그 — 예상 300ms → 실제 95초 순차 루프 내 sleep_for(batchIndex × 3ms) 반복 호출 → worker 31 기준 ~92초 낭비.
→ sleep_until(testStart + batchIndex × 3ms) 절대시각 기준으로 변경.
